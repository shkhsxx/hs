# 출시 전 의사결정 요청서

> 프로젝트: SecondBoost
> 작성 기준: PRD 분석 결과 (15개 케이스, Conflict 2건)
> 목적: 개발 착수 전 Product Owner가 결정해야 할 항목을 임팩트·긴급도 기준으로 우선순위화
> 수신: Product Owner / 창업자

---

## 리스크 등급 기준

본 문서는 미결 항목을 아래 3등급으로 분류한다.

| 등급 | 정의 | 결정 기한 |
|------|------|-----------|
| 🔴 **Blocker** | 결정 없으면 해당 기능 개발 착수 불가. 구현 방식이 결정에 완전 종속되거나 법적·보안 리스크 존재 | 개발 착수 전 |
| 🟡 **주의** | 결정 지연 시 일정 또는 기능 품질에 직접 영향. 다른 기능과 연쇄 영향 가능 | 스프린트 내 |
| 🟢 **참고** | 출시 후 대응 가능. 단, 사전 결정 시 UX 일관성 확보에 유리 | 출시 전 |

### 우선순위 판단 매트릭스

```
        높음 ┌─────────────┬─────────────┐
             │             │             │
   임 팩 트  │  🟡 주의   │ 🔴 Blocker │
             │             │             │
        낮음 ├─────────────┼─────────────┤
             │  🟢 참고   │  🟡 주의   │
             │             │             │
             └─────────────┴─────────────┘
                  낮음           높음
                        긴급도
```

- **임팩트:** 미결 시 영향받는 기능 수 및 범위 (전체 서비스 흐름 vs 특정 기능 내부)
- **긴급도:** 결정이 필요한 시점 (개발 착수 전 필수 vs QA 전 vs 출시 후 가능)

## 🔴 Blocker — 개발 착수 전 반드시 결정

### B-01. 회원가입 인증 방식
**관련 케이스:** C14 (Conflict)
**임팩트:** 전체 서비스 접근의 시작점. 인증 방식에 따라 회원가입, 로그인, 보안 아키텍처 전체가 달라짐
**긴급도:** 최우선 — 인증 모듈 없이 다른 기능 개발 불가

| 구분 | 내용 |
|------|------|
| 요구 A | 창업자 (Slack DM): 이메일 + 매직링크 단독 인증, 비밀번호 없음 |
| 요구 B | 보안 컨설턴트 (CS 이메일): 비밀번호 + OTP 2단계 인증 필수 — 금융 데이터 보안 감사 통과 조건 |
| 미결 시 리스크 | 보안 감사 미통과 시 서비스 출시 자체가 불가. 또는 매직링크 방식으로 개발 후 전면 재설계 |
| 결정 요청 | **PO가 보안 감사 기준 확인 후 인증 방식 최종 결정** |

---

### B-02. 게시글 자동 저장 위치
**관련 케이스:** C15 (Conflict)
**임팩트:** 저장 위치에 따라 API 설계, DB 스키마, 프론트엔드 구현 방식이 완전히 다름
**긴급도:** 최우선 — 동일 기능을 두 방식으로 동시 개발 불가

| 구분 | 내용 |
|------|------|
| 요구 A | 기획자B + 창업자 초기: 서버 저장 → 다기기 이어쓰기 지원 |
| 요구 B | 창업자 번복: 로컬(브라우저) 저장 → 트래픽·비용 절감 |
| 미결 시 리스크 | 개발팀이 임의로 선택하면 출시 후 전면 재설계 가능성 |
| 결정 요청 | **PO가 다기기 지원 여부와 서버 비용 수용 범위를 고려하여 저장 위치 결정** |

---

### B-03. 탈퇴 후 개인정보 보관 기간
**관련 케이스:** C08
**임팩트:** 개인정보보호법 위반 시 과태료·서비스 중단 가능. DB 삭제 배치 설계에 직접 영향
**긴급도:** 출시 전 법적 검토 필수

| 구분 | 내용 |
|------|------|
| 현황 | 탈퇴 기능 UI는 구현 가능하나, 탈퇴 후 데이터 처리 로직 설계 불가 |
| 미결 시 리스크 | 즉시 삭제 vs 보관 기간 설정에 따라 DB 설계, 배치 로직, 유저 공지 문구 모두 달라짐 |
| 결정 요청 | **법무팀 또는 개인정보보호 전문가와 협의 후 보관 기간 확정** |

## 🟡 주의 — 스프린트 내 결정 권장

### W-01. 다크모드 적용 범위
**관련 케이스:** C12
**임팩트:** 전체 적용 시 모든 화면 컴포넌트 재작업. 일부 적용 시 일관성 문제 발생 가능
**결정 요청:** PO가 이번 분기 출시 일정과 디자인 리소스를 고려하여 전체 vs 일부 확정

---

### W-02. 실시간 채팅 — 외부 솔루션 vs 자체 개발
**관련 케이스:** C13
**임팩트:** 외부 솔루션 vs 자체 개발에 따라 구현 방향 전체가 달라짐
**결정 요청:** PO가 비용 검토 결과를 바탕으로 방향 결정 (원문: "외부솔루션 쓸지 자체개발할지 결정 안남 — 비용검토 후")

---

### W-03. 구독 정지 후 서비스 접근 범위
**관련 케이스:** C06
**임팩트:** 결제 실패 유저의 경험과 재구독 전환율에 직접 영향
**결정 요청:** 완전 차단 / 읽기 전용 / 일정 기간 유예 중 방향 선택

---

### W-04. 다른 팀 소속 유저 초대 처리
**관련 케이스:** C05
**임팩트:** 멀티팀 사용자 데이터 모델 설계에 영향. 초대 수락 플로우 분기 필요
**결정 요청:** 팀 이동 / 중복 소속 허용 / 초대 거절 중 선택

---

### W-05. 알림 off 유저의 중요 주문 알림 처리
**관련 케이스:** C02
**임팩트:** 알림 시스템 예외 로직 설계에 영향. 유저 신뢰도와 직결
**결정 요청:** 알림 off여도 중요 알림은 강제 발송할 것인지, 또는 알림 off를 절대적으로 존중할 것인지 결정

---

### W-06. 온보딩 튜토리얼 노출 횟수
**관련 케이스:** C11
**임팩트:** 유저별 노출 상태 저장 로직 및 다시보기 트리거 설계에 영향
**결정 요청:** 최초 1회만 / 앱 재설치 시 재노출 / 설정 초기화 시 재노출 중 정책 선택

---

### W-07. 장바구니 삭제 전 알림 여부
**관련 케이스:** C03
**임팩트:** 알림 시스템 연동 여부에 따라 추가 개발 공수 발생
**결정 요청:** 알림 제공 여부 및 제공 시 방식(푸시/이메일) 결정 (원문: "있으면 좋고 없어도 되고 그건 알아서")

---

### W-08. 쿠폰 최소 주문 금액 조건
**관련 케이스:** C09
**임팩트:** 쿠폰 적용 검증 로직 분기 발생. 결제 화면 UX 설계에 영향
**결정 요청:** 사내 논의 완료 후 최소 금액 조건 적용 여부 및 금액 확정

---

### W-09. 탈퇴 후 동일 이메일 재가입 정책
**관련 케이스:** C08
**임팩트:** 계정 DB 설계, 이메일 중복 검사 로직에 영향
**결정 요청:** 재가입 허용 여부 및 허용 시 가능 시점(즉시 / 일정 기간 후) 결정

---

### W-10. 장바구니 만료 시각 기준
**관련 케이스:** C03
**임팩트:** 자정 기준 vs 정확한 시각 기준에 따라 유저 경험 차이 발생
**결정 요청:** 자정 처리 vs 담은 시각 기준 7×24시간 중 선택

---

### W-11. 리뷰 사진 첨부 제한 기준
**관련 케이스:** C07
**임팩트:** 제한 기준 미결 시 업로드 기능 완성 불가
**결정 요청:** 리뷰 1건당 최대 사진 수, 파일 최대 용량(MB), 부적절 사진 처리 정책 결정 (원문: "알아서 적절히 제한 걸어주시면 됩니다")

---

### W-12. 대용량 거래 내역 다운로드 처리 방식
**관련 케이스:** C10
**임팩트:** 처리 방식 미결 시 대용량 구간 구현 착수 불가
**결정 요청:** 1만 건 이상 다운로드 처리 방식 및 최대 허용 건수 결정 (기술팀 협의 필요) (원문: "정확히 몇 건까지 한 번에 받을 수 있어야 하는지는 기술팀이랑 정해야 할 것 같습니다")
